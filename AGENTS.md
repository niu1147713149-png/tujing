# 图鲸项目本地规范

本文件作用域：当前目录及其全部子目录。

## 必须遵守的本地化规范
- 面向用户的文案默认使用**简体中文**。
- 不要把用户可见文本写成乱码、问号或半乱码占位。
- 修改页面、组件、接口返回文案后，必须检查用户可见文本是否仍为正常简体中文。
- 若当前终端/脚本环境可能污染中文，优先使用**UTF-8 明确写入**或 Unicode 转义来保证文本稳定。

## 编码与字符安全规范
- 所有源码文件统一按 **UTF-8** 处理。
- 避免把高风险字符直接写入用户界面文本，优先使用稳定字符：
  - 用 `/` 代替 `·`
  - 用 `x` 代替 `×`
  - 谨慎使用 `•`, `✦`, `→` 等特殊符号
- 如果某个特殊字符不是必须，就不要为了“好看”引入它。
- 若发现文件里出现 `?`, `??`, `???` 或明显乱码，视为缺陷，必须继续修复，不要带着提交。

## 交互与产品规范
- 生成图片、任务状态、错误提示等用户可见状态文案，必须明确且可理解。
- 不要让用户误以为“请求没发出去”；超时场景要明确说明：请求是否已发出、是否正在等待上游、是否可能后台继续生成。
- 新增功能时，优先保持现有中文体验完整，不要只改功能不改文案。
- 涉及页面、组件、API、任务流、设计语言、测试策略的开发，必须同时遵守 `docs/PROJECT_STANDARDS.md`。

## Subagent 协作规范
- 对于中大型代码任务，默认优先使用 Codex native subagents 做并行分析、实现或验证。
- 除非任务极小，或没有独立可拆分子任务，否则不要跳过 subagent。
- 进入实现前，优先判断是否存在可并行的分析、修复、验证子任务；若存在，应先分派 subagent 再继续主流程。
- 若任务包含多个相对独立的文件、模块、问题点或验证项，优先采用 subagent 并行推进。
- 若因任务过小或强耦合而未使用 subagent，需要先确认确实不存在合理拆分空间。

## 自动化测试规范
- 只要任务涉及页面交互、前端报错复现、接口联调、端到端验证或“你去测一下”这类请求，默认优先调用 `playwright-cli` 技能做自动化测试，不需要等待用户再次提醒。
- 发起本地 Playwright 测试前，必须先检查并清理 5173 / 8000 端口上的旧前端或后端进程，避免命中历史会话、错误端口漂移或代理假 500。
- 本地测试优先验证：页面是否可打开、`/api/tasks` 是否可达、生成按钮是否能创建任务、结果页是否进入轮询状态。
- 若测试失败，必须同时收集页面弹窗、浏览器控制台、网络请求和后端日志，再进入修复。

## 修改后的最低检查
每次涉及用户可见内容的修改后，至少检查：
1. 首页
2. 生成页
3. 结果页
4. 相关 API 返回错误文案

如果任一页面仍有乱码或问号，任务不算完成。
