# 图鲸项目规范文档

> 目的：统一图鲸项目的设计语言、前后端开发方式和测试标准，减少后续沟通成本，并让 Codex / 其他开发者默认按同一套规则工作。

---

## 1. 适用范围

本规范适用于当前仓库全部业务开发，覆盖：

- 前端页面与组件
- 后端 API 与任务流
- 文案、本地化与编码
- 测试、验收与提交前检查

当 README、开发说明、临时方案文档与本规范冲突时，**以本规范为准**。

---

## 2. 产品定位

图鲸是一个面向电商场景的 AI 出图工具，当前阶段为 **MVP / Task-based generation flow**。

核心目标：

1. 用户能够快速选择模板并输入 prompt
2. 系统创建生成任务，而不是同步长等待
3. 结果页通过轮询展示任务状态
4. 用户能看懂当前状态、失败原因和下一步操作

当前不追求“大而全”，优先级为：

**稳定 > 清晰 > 一致 > 炫技**

---

## 3. 设计语言总则

### 3.1 设计方向

图鲸的视觉基调必须保持：

- **深色基调**
- **高级感 / 工具感 / 克制感**
- **Linear 主基调**
- 结果图优先于装饰元素

不要做成：

- 廉价 demo 风
- 过度营销风
- 组件堆砌感强的草台后台页

### 3.2 一致性原则

所有页面必须共享统一的：

- 颜色体系
- 圆角体系
- 边框强弱
- 字体层级
- 间距节奏
- 状态表达方式

**禁止每个页面自行发明一套 UI。**

---

## 4. 前端 UI 规范

### 4.1 页面结构规范

当前主要页面：

- `/` 首页
- `/generate` 生成页
- `/result/[taskId]` 任务结果页

每个页面都应具备：

1. 页面主标题
2. 页面说明文案
3. 当前步骤或当前状态
4. 明确的主操作按钮
5. 异常状态和空状态

### 4.2 页面角色划分

#### 首页
职责：

- 品牌表达
- 模式说明
- 配置检测入口
- 进入生成工作台

首页不是配置页，也不是结果页，不应堆积过多业务细节。

#### 生成页
职责：

- 选择模板
- 选择或编辑 prompt
- 创建任务

生成页必须像“工作台”，不是普通表单页。

#### 结果页
职责：

- 显示任务状态
- 显示结果图
- 提供下载、再来一张、返回修改提示词

结果页必须让用户一眼看出：

- 现在是否还在生成
- 请求是否已经发出
- 是成功还是失败
- 失败后该怎么办

### 4.3 组件规范

组件应遵守：

- 单一职责
- 文案尽量由页面层控制
- 公共模板数据集中在 `components/template-options.ts`
- 视觉基础组件应复用全局 class 规则，不得各自随意拉风格

当前关键组件：

- `TemplateSelector`
- `GeneratingLoader`
- `ResultDisplay`
- `ImageUploader`

### 4.4 视觉层级规范

#### 色彩
- 主背景：近黑
- 面板背景：弱亮度差异
- 主强调色：紫蓝系
- 成功色：绿色，只用于状态
- 错误色：红色，只用于错误

#### 边框
- 优先使用弱边框
- 不用厚重描边
- 不要到处高对比边框

#### 阴影
- 阴影用于层级，不用于炫效果
- 面板允许柔和阴影
- 不允许大量夸张发光

#### 圆角
- 页面主卡片：统一中大圆角
- 子卡片：统一中圆角
- 不允许同一页面同时混用大量不同圆角体系

### 4.5 文案规范

用户可见文案必须：

- 默认简体中文
- 简洁直接
- 状态明确
- 不用半专业、半口语、半乱码表达

例如：

#### 好文案
- 正在生成图片
- 请求已发出，正在等待模型返回
- 任务创建成功
- 返回修改提示词

#### 差文案
- 正在处理中...
- 系统繁忙请稍后
- 生成异常
- ??? / 乱码

### 4.6 字符安全规范

为避免编码污染，用户界面文字优先使用稳定字符：

- 用 `/` 代替 `·`
- 用 `x` 代替 `×`
- 谨慎使用 `•`, `✦`, `→`

如果某个特殊字符不是必须，就不要引入。

## 5. 前端工程规范

### 5.1 技术约束

- 使用 React 函数组件 + Hooks
- 使用 TypeScript
- 样式以 Tailwind 为主
- 页面状态优先就近管理
- 不轻易增加依赖

### 5.2 代码组织

推荐边界：

- `app/`：页面与 API 路由
- `components/`：页面组件与复用组件
- `lib/`：业务逻辑、任务流、配置、模型调用
- `config/`：本地配置模板
- `docs/`：规范与设计文档

### 5.3 命名规范

- 组件：PascalCase
- 函数：camelCase
- 常量：UPPER_SNAKE_CASE
- 存储 key：统一 `tujing.*`
- 任务 ID：统一 `task_*`

---

## 6. 后端开发规范

### 6.1 架构原则

图鲸当前后端必须坚持 **任务型生成**：

- 先创建任务
- 再异步执行
- 前端轮询状态

不要重新回到“前端一直挂着等一个长请求”的模式。

### 6.2 分层规范

后端逻辑必须分层：

#### API 路由层
位置：`app/api/*`

职责：

- 参数接收
- 状态码控制
- 响应格式统一

禁止：

- 在 route 文件里堆大量核心业务逻辑

#### 任务存储层
位置：`lib/task-store.ts`

职责：

- 创建任务
- 查询任务
- 更新任务状态

#### 任务执行层
位置：`lib/task-runner.ts`

职责：

- 处理 queued -> processing -> succeeded / failed
- 控制单任务执行

#### 模型调用层
位置：`lib/gemini.ts`

职责：

- 统一封装 Gemini 请求
- 统一 timeout / retry / 结果解析

#### 配置层
位置：`lib/ai-config.ts`

职责：

- 读取配置文件
- 校验 domain / apiKey / modelId

### 6.3 接口规范

当前任务流接口：

- `POST /api/tasks`
- `GET /api/tasks/[taskId]`
- `POST /api/tasks/[taskId]/regenerate`
- `POST /api/config-check`

兼容保留接口：

- `POST /api/generate`
- `POST /api/upload`

### 6.4 返回规范

所有失败响应必须：

- 给出明确中文错误信息
- 区分用户可理解错误和系统错误

所有关键请求尽量包含：

- `requestId`
- `taskId`
- `status`

### 6.5 超时规范

超时文案必须明确表达：

1. 请求是否已发出
2. 是否在等待上游
3. 是否可能后台继续生成

禁止只返回模糊错误。

### 6.6 存储规范

当前阶段允许使用文件型任务存储。  
后续升级 SQLite 时，必须：

- 保持 `task-store` 抽象边界
- 不把数据库逻辑散落在 route 里

## 7. 测试规范

### 7.1 最低验证要求

每次涉及功能或文案变更后，至少验证：

1. 首页 `/`
2. 生成页 `/generate`
3. 结果页 `/result/[taskId]`
4. 配置检测接口
5. 任务创建接口
6. 任务查询接口
7. 再来一张接口

### 7.2 最低命令检查

```bash
npm run build
npx tsc --noEmit
```

### 7.3 任务流专项检查

必须验证以下状态：

- queued
- processing
- succeeded
- failed

必须验证以下行为：

- 创建任务后返回 taskId
- 结果页能轮询状态
- 成功后能展示图片
- 失败后能展示明确错误
- 再来一张会创建新任务，而不是覆盖旧任务

### 7.4 文案与编码检查

任何下面情况都视为阻塞问题：

- `?`
- `??`
- `???`
- 乱码
- 半乱码

不能只看源码，必须看最终网页渲染结果。

## 8. Codex 开发约束

为了减少后续沟通，Codex 后续修改必须默认遵守：

1. 优先保持设计语言一致
2. 不擅自引入全新页面风格
3. 不擅自回退为同步长请求架构
4. 不得把乱码、问号文案带入结果
5. 修改用户可见内容后，必须自检首页/生成页/结果页/API 文案

如果某次修改不能同时满足功能和规范，优先保证：

**规范正确性 > 视觉小优化 > 新花样**

## 10. 当前执行原则

- 先稳定，再扩展
- 先一致，再美化
- 先任务流，再高级功能
- 先中文正确，再视觉精修
- 发现可见乱码，必须继续修，不得忽略
