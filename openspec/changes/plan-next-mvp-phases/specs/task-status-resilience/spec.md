## ADDED Requirements

### Requirement: 结果页必须清楚解释异步任务状态
系统 MUST 在结果页使用清晰的简体中文说明任务当前处于排队、生成中、成功、失败或超时后的状态，避免让用户误以为请求没有发出。

#### Scenario: 任务仍在处理中
- **WHEN** 任务状态为 queued 或 processing
- **THEN** 结果页 MUST 明确说明请求已经发出、当前正在等待上游模型返回，并继续展示轮询中的进度信息

#### Scenario: 任务失败或超时
- **WHEN** 任务进入 failed 状态或命中上游超时语义
- **THEN** 系统 MUST 显示可理解的失败原因，并提供重新生成或返回修改提示词的后续动作

### Requirement: 后端必须保留任务上下文并支持恢复
系统 SHALL 在任务恢复或重新生成时保留任务所属订单、提示词、模板和模型上下文，避免用户重复配置。

#### Scenario: 服务重启后恢复未完成任务
- **WHEN** 后端启动时发现 queued 或 processing 的未完成任务
- **THEN** 系统 MUST 将这些任务重置到可恢复状态并继续执行，直到任务进入 succeeded 或 failed

#### Scenario: 基于现有任务重新生成
- **WHEN** 用户从结果页触发重新生成
- **THEN** 新任务 MUST 继承原任务的订单、提示词、模板和模型设置

### Requirement: 批量任务必须支持部分成功场景
系统 SHALL 在批量生成中正确区分成功与失败结果，并允许用户在全部完成前后理解当前批次状态。

#### Scenario: 批量生成部分成功
- **WHEN** 一个批次中部分任务成功、部分任务失败
- **THEN** 结果页 MUST 同时展示成功数量、失败数量和当前总进度，并保留成功结果的下载入口
