## Context

当前仓库已经从旧的一体化 Next.js 实现收口为 `frontend/` + `backend/` 双目录架构，核心主线是“创建订单 -> 进入工作台 -> 创建生成任务 -> 结果页轮询 -> 历史回看 / 再生成”。近期代码已经开始补齐订单能力与模型选择能力，但后续开发如果没有统一规划，很容易再次出现以下问题：

- 页面与接口各自推进，用户状态文案不一致
- 后端任务恢复、超时语义与前端展示脱节
- 规范强调的简体中文、任务制流程、最低检查项没有被系统化纳入开发节奏

约束条件：
- 继续采用异步任务制，不回退到同步长等待生成
- 用户可见文案必须保持简体中文与 UTF-8 安全
- 优先复用现有 React / FastAPI / SQLite 结构，不新增依赖
- 后续规划必须与 `docs/PROJECT_STANDARDS.md` 一致

## Goals / Non-Goals

**Goals:**
- 用 OpenSpec 固化图鲸后续开发的能力边界与交付顺序。
- 先补齐订单工作流、结果状态表达、发布护栏，再考虑扩展更多模板或业务形态。
- 让每个阶段都能对应到明确页面、接口、测试与验收点。

**Non-Goals:**
- 本次不直接实现所有功能代码。
- 本次不引入账号体系、支付、多人协作或全新任务编排系统。
- 本次不推翻当前深色工具化设计语言，也不重做整体品牌方向。

## Decisions

### 1. 按能力域规划，而不是按页面零散排期
- 选择：把后续开发分成“订单中心工作流”“任务状态韧性”“发布就绪护栏”三个能力域。
- 原因：当前问题跨越首页、生成页、结果页、历史页和后端服务，单页排期会遗漏共享状态与接口边界。
- 备选方案：按页面拆分首页 / 生成页 / 结果页任务。
- 放弃原因：难以覆盖共享任务流与统一验收标准。

### 2. 先做状态清晰与恢复能力，再扩展新功能面
- 选择：优先保证用户能看懂“请求是否发出、现在在等什么、失败后怎么办、是否可继续恢复”。
- 原因：图鲸当前是任务制产品，状态解释与恢复链路直接决定可用性。
- 备选方案：优先扩充更多模板或模型。
- 放弃原因：如果状态表达和恢复机制不稳定，新增能力只会放大混乱。

### 3. 把发布护栏纳入产品规划，而不是仅靠人工记忆
- 选择：将配置可用性、核心接口烟测、页面中文文案检查写入能力要求与任务清单。
- 原因：当前项目强依赖本地联调与上游模型，若没有护栏，回归风险高且排查成本大。
- 备选方案：仅在开发完成后做一次人工联调。
- 放弃原因：不可重复，也无法稳定支持后续增量迭代。

## Risks / Trade-offs

- [风险] 订单能力仍在演进，当前代码与规划可能短期出现偏差 -> [缓解] 先用 OpenSpec 锁定目标行为，再按阶段更新实现。
- [风险] 上游模型超时或失败语义不稳定 -> [缓解] 优先定义前端统一提示文案与后端错误映射策略。
- [风险] 仓库已有较多未提交变更，规划容易与当前 WIP 冲突 -> [缓解] 本次仅新增 OpenSpec 规划文件，不改动现有业务代码。
- [风险] 无新增依赖会限制可观测性方案 -> [缓解] 优先使用现有日志、健康检查、烟测脚本与页面态提示达成最低可用。

## Migration Plan

1. 先完成本次 OpenSpec proposal / design / specs / tasks，作为后续迭代入口。
2. 第一阶段按任务清单补齐订单与结果闭环，并同步新增或更新自动化验证。
3. 每完成一个能力域，就回填验证结果，再决定是否继续创建新的 OpenSpec change 扩展更多能力。
4. 全部任务落地并验证通过后，再归档本 change，并把稳定能力沉淀到 `openspec/specs/` 基线。

## Open Questions

- 订单维度是否还需要引入更多筛选字段，例如客户、渠道或模板标签？
- 超时任务未来是仅提示用户手动重试，还是要增加受控自动重试？
- 结果页是否需要提供“部分成功先下载，其余继续等待”的更显式交互？
